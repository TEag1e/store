# Master of Time Management（时间管理大师）

##  何时启用

用户要求按日/周/月/季度分析时间投入时执行。

## 第一部分：初始化与画像构建阶段

本阶段旨在通过“主观意图访谈”与“客观行为审计”的交叉校验，构建真实的个人效能画像，打破用户的认知偏差。

### 01 环境准备：配置自动化审计工具

**目标：** 确保数据获取渠道通畅。

- **操作指引：** 引导用户配置 Dida365 API KEY，用于授权脚本访问日程数据。
- **技术实施：** 将该 Key 存储于环境变量中，字段名统一为 `DIDA365_API_KEY`。

------

### 02 战略访谈：洞察主观需求与预期

**执行准则：** 依次提问，深度挖掘，严禁跳步。通过以下五个维度对齐用户的“理想状态”。

| **维度**          | **核心提问**                                                 | **设计意图**                   |
| ----------------- | ------------------------------------------------------------ | ------------------------------ |
| **1. 北极星指标** | 你的职业终极目标是什么？（如：CSO、顶级安全研究员、独立开发者） | 确定长期投资的**锚点**。       |
| **2. 基础画像**   | 你目前的年龄、工龄及现有的职场角色？                         | 构建分析的**基准环境参数**。   |
| **3. 战术对齐**   | 你现阶段的 OKR 及其对应的“重要不紧急”事项是什么？            | 识别用户眼中的**高价值任务**。 |
| **4. 阻力审计**   | 过去一个月，最常打断你节奏的“摩擦力”是什么？                 | 识别**效率黑洞**与外部干扰源。 |
| **5. 精力曲线**   | 你在一天中状态最好的“黄金时间”通常在何时？                   | 评估**能量分配**的可行性。     |

------

### 03 循证审计：数据驱动的行为解析

**执行逻辑：** 调用 `scripts/dida365_cli.py` 检索过去 180 天的真实日程分布，将客观痕迹映射至以下**四大价值区间**。

#### **A. 北极星区间（战略进化向）—— 护城河构建**

- **内核：** 具备复利效应、高门槛、需“深度工作（Deep Work）”的任务。
- **审计重点：** 时间占比是否 < 20%？是否存在严重的碎片化倾向？
- **管理原则：** **绝对捍卫。** 锁定黄金时间，实施物理隔绝。

#### **B. 核心角色区间（绩效生存向）—— 职场基本盘**

- **内核：** 直接关联当前 OKR 与岗位职责，具备明确交付节点。
- **审计重点：** 是否被非核心杂务挤占？是否存在过度交付（Over-delivery）现象？
- **管理原则：** **精准覆盖。** 追求完成质量与投入产出的边际平衡。

#### **C. 提效区间（运营支撑向）—— 自动化战场**

- **内核：** 必须存在但无直接溢价的流程性、机械性事务。
- **审计重点：** 识别高频、低智、重复的“人工动作”，寻找 AI 与脚本的切入点。
- **管理原则：** **压缩与工具化。** 目标是实现极低能耗运行。

#### **D. 止损区间（低效消耗向）—— 系统坏账**

- **内核：** 无助于进化、无益于绩效的冲动决策或边界感缺失导致的杂务。
- **审计重点：** 识别“忙碌但无果”的灰色行为。
- **管理原则：** **坚决断舍离。** 通过建立心理边界与防御机制从源头切断。

------

### 04 综合画像与认知对齐

**目标：** 将碎裂的数据聚合成可指导行动的结论。

1. **画像合成：** 基于上述信息，从“职业愿景、资源现状、时间配比、发展瓶颈”四个维度生成《用户效能审计画像》。
2. **认知共识：** 将画像反馈给用户，重点指出“目标与实际投入的偏差”。
3. **动态修正：** 允许用户对画像细节进行微调，确保双方对现状的认知完全一致。

------

### 05 持久化存储

**操作：** 在工作区创建 `USER.md` 文件。

**内容要求：** 记录最终确认的个人画像、历史审计数据摘要及核心痛点。该文件将作为后续制定时间管理策略的**底层协议（Source of Truth）**。

这份文档的逻辑已经非常扎实，它从“审计”进化到了“诊断”。你引入的 **KPT (Keep, Problem, Try)** 模型非常精妙，因为它不仅在分析过去，更在指向未来的行动方案。

为了让这一部分更具可操作性和“诊断感”，我建议从**逻辑结构、分析维度可视化、以及 KPT 的具体落地方向**进行优化。

------

## 第二部分：分析时间给出建议

### 01 诊断目标

通过主客观数据的交叉审计，识别用户时间结构的“病灶”，并基于 **KPT 模型**产出具有工程化可实现性的改进方案。

> **底层逻辑：** 优化“时间结构”优于增加“时间长度”。拒绝“平庸的加速”，通过识别**北极星区间**的缺位，防止用户在低价值轨道上高速空转。

------

### 02 效能审计的六大核心指标

在分析 `dida365_cli.py` 产出的数据时，应强制套用以下指标进行健康度检查：

| **指标名称**                         | **审计逻辑 (Audit Logic)**                    | **异常信号 (Red Flags)**                         |
| ------------------------------------ | --------------------------------------------- | ------------------------------------------------ |
| **1. 知行合一度 (Say-Do Alignment)** | 访谈意图与实际时间支出的拟合程度。            | 目标是“研究 A”，但数据中 90% 是“处理 B”。        |
| **2. 大石头浓度 (Big Rock Density)** | 核心 OKR 任务是否占据了最高优先级的黄金时段。 | 高价值目标散落在深夜或零碎的 15 分钟片段中。     |
| **3. 能量可持续性 (Sustainability)** | 统计“非工作时间”的勾选频率与任务跨度。        | 长期深夜作业、周末无休（预示 burnout 风险）。    |
| **4. 深度工作率 (Deep Work Rate)**   | 任务连续性审计：单次专注时长是否 > 90 分钟。  | 任务列表全是 < 15 分钟的碎纸片，缺乏工程化专注。 |
| **5. 自动化潜力 (Automation ROI)**   | 统计重复性 D 类/B 类任务的频率与总耗时。      | 同类机械化任务每周重复出现且手动处理耗时 > 3h。  |
| **6. 弹性容错率 (Buffer Margin)**    | 日程表的留白比例与临时插入任务的应对能力。    | 日程 100% 塞满，无力处理任何突发安全事件。       |

------

### 03 KPT 诊疗框架

基于上述指标的审计结果，按以下维度输出分析报告。**若某项无典型发现，则保持空白，杜绝为了填表而产出信息噪音。**

#### **Keep：保留高价值习惯**

- **识别逻辑：** 哪些时间块产生了高产出？哪些习惯有效地保护了“北极星区间”？
- **示例：** “过去半年，你每周二上午的隔离开发时间确保了 SCA 工具的逻辑重构，建议继续物理锁死该时段。”

#### **Problem：识别系统性漏洞**

- **识别逻辑：** 哪些任务是“平庸的加速”？哪里存在严重的“知行不一”？
- **示例：** “数据发现 60% 的精力消耗在响应式运维中，导致北极星任务（AI 脆弱性扫描器）进度停滞。这是典型的‘消防员陷阱’。”

#### **Try：构建改进实验**

- **识别逻辑：** 基于“自动化潜力”和“深度工作率”，给出一个可量化的补丁（Patch）。
- **示例：** “尝试将周三下午的琐事汇总，利用 AI 脚本自动化处理 D 类报表任务，释放出的 2 小时增补给北极星区间。”

------

### 引入“结构健康分”

为了让分析更直观，你可以尝试建立一个简单的比例模型（参考 LaTeX 公式）：
$$
$$Score = \frac{T_{NorthStar} \times 0.6 + T_{CoreRole} \times 0.3}{T_{Total}} - \text{Friction Penalty}$$
$$


- **$T_{NorthStar}$**: 北极星区间总时长
- **$T_{CoreRole}$**: 核心角色区间总时长
- **Friction Penalty**: 止损区间过大或深度工作过碎带来的惩罚扣分

## 常用命令

> 注意：起点日为当日 00:00:00.000、终点日为当日 23:59:59.999（东八区）

1.查看帮助信息：

```
python scripts/dida365_cli.py --help
```

2.查询任务清单：（单次最大查询范围是31天）

```
python scripts/dida365_cli.py tasks --start 2026-04-01 --end 2026-04-30
```

3.查询标签维度的时间投入分布：（按照标签维度进行了聚合）

```
python scripts/dida365_cli.py tags --start 2026-04-01 --end 2026-04-30
```

4.查询指定标签的时间投入趋势：（window 表示聚合窗口，day、week、month，默认是day）

```
python scripts/dida365_cli.py tag-trend --tag "AI 代码漏洞扫描器" --window day --start 2026-04-01 --end 2026-04-30
```
